## 2026-03-10 - [Added @Immutable to Library UI Data Classes] **Learning:** [Jetpack Compose marks data classes containing collections like `List` as unstable, which can cause severe recomposition issues in large lists like LibraryScreen. `PersistentList` alone might not be enough if the class itself isn't marked.] **Action:** [Always annotate data classes used in Lazy columns/grids with `@Immutable` if they contain collections or interface types to prevent performance bottlenecks.]
## 2026-03-10 - [Optimize Manga Backup History Queries] **Learning:** [When backing up manga history, the app was executing a DB query per history entry to retrieve the chapter URL. This created an N+1 query bottleneck within the database transaction.] **Action:** [Pre-fetch all chapters for a given manga with a single DB query, map them by ID, and use that map to assign URLs to history entries, thereby resolving the bottleneck.]
## 2026-03-10 - [Avoid Synchronous SharedPreferences Reads inside DefaultGetResolver] **Learning:** [Calling SharedPreferences repeatedly inside `mapFromCursor` of a database `GetResolver` causes massive N+1 performance bottlenecks during large queries (e.g., >1000 items). Caching the preference in the resolver avoids this, but requires avoiding singletons to prevent state corruption.] **Action:** [Always instantiate `DefaultGetResolver` classes dynamically per-query if they cache external state, and pre-fetch or lazily fetch necessary SharedPreferences once per query instance.]
## 2026-03-15 - [Avoid Intermediate Lists with mapNotNull] **Learning:** Using `.filter {}.map {}` chains on large lists allocates an intermediate list, which increases GC overhead and reduces performance on critical paths like resolving the next chapter. **Action:** Replace `.filter {}.map {}` chains with `.mapNotNull { if (...) ... else null }` to avoid intermediate list allocations and improve performance, especially when quickly jumping to reading.
## 2026-03-15 - [Optimize String Splitting in DB Cursor Mapping] **Learning:** [Database queries returning grouped concatenated strings (e.g. `GROUP_CONCAT` in SQLite) can cause massive CPU and GC overhead when mapped to UI models if the raw string is indiscriminately `split()` inside a tight loop.] **Action:** [Use zero-allocation fast paths with `indexOf` loops to calculate count when complex splitting/filtering isn't strictly necessary, and `String.contains` when simple substring existence checks are enough to prevent performance bottlenecks.]
## 2026-03-19 - Prevent Redundant State Updates in ViewModels
**Learning:** Flow `changes()` from preferences without `distinctUntilChanged()` causes redundant state updates and UI recompositions when preferences emit the same value.
**Action:** When observing preferences, use `changes().observeAndUpdate(viewModelScope)` instead of `changes().collect { ... }` to automatically apply `distinctUntilChanged()` and handle lifecycle collection efficiently.
## 2026-03-19 - Optimize LibraryUpdateJob List Operations
**Learning:** Chaining `.filter {}.filter {}.map {}` on collections creates multiple intermediate lists, which is inefficient.
**Action:** Use `.mapNotNull {}` to combine filtering and mapping in a single pass to reduce memory allocations and improve performance, especially when handling large datasets like chapter updates.

## 2026-03-23 - [Avoid Redundant Updates from Combined UI Settings Flows]
Learning: Combining multiple UI preference flows (like `gridSize` and `layout`) without applying `.distinctUntilChanged()` creates a stream that emits redundantly when upstream preferences change, causing redundant downstream state recalculations (e.g. `libraryScreenState`).
Action: Always append `.distinctUntilChanged()` after `combine()` blocks that aggregate multiple UI preferences before merging them into the main screen `StateFlow` to prevent spamming the UI with redundant updates.
## 2026-03-27 - [Defer ExpressivePicker Scroll State Reads] **Learning:** [Directly reading `LazyListState.firstVisibleItemIndex` inside a `LazyColumn` item block or lambda (like `itemsIndexed`) forces recomposition of those items on every pixel of scroll since the visible index constantly shifts.] **Action:** [Use `derivedStateOf { listState.firstVisibleItemIndex }` to defer reading the scroll state, calculating the desired index target or property outside the raw composition phase, which drastically improves scroll performance.]
