## 2026-03-19 - Prevent Leaks by Removing GlobalScope in TrackingSyncJob
**Learning:** `launchIO { ... }` from `eu.kanade.tachiyomi.util.system` uses `GlobalScope`, which can easily cause memory and resource leaks. In `TrackingSyncJob`, calling `launchIO { context.notificationManager.cancel(...) }` in the `finally` block creates an unmanaged coroutine that could continue running or leak references after the work is done.
**Action:** Replace `launchIO` with a safe, scoped Coroutine invocation (like `withContext(Dispatchers.IO)`) to ensure the cleanup logic is properly bounded and executed synchronously before `doWork()` returns, preventing potential memory leaks and unmanaged work.
## 2026-03-24 - [Avoid N+1 StorIO queries with bulk fetches] **Learning:** Executing `db.getManga(it.toLong()).executeAsBlocking()` within `mapNotNull` on list inputs in background jobs (like `FollowsSyncProcessor.toMangaDex`) creates an N+1 query problem that blocks the execution thread. **Action:** Replace looping synchronous StorIO database queries on collections with chunked batch queries, e.g., `.chunked(900).flatMap { db.getMangas(it).executeAsBlocking() }` to improve query performance and reduce lock contention.
## 2026-03-25 - [Bulk Fetch Tracks in Background Sync] **Learning:** Fetching tracks individually for every manga inside the loop in `TrackSyncProcessor.process()` creates an N+1 query problem, severely blocking the execution thread during large tracking syncs. **Action:** Refactor tracking sync processing to pre-fetch all needed tracks using `.chunked(900).map { async { db.getTracks(it).executeOnIO() } }.awaitAll().flatten()` and mapping them by manga ID, avoiding individual SQL queries inside loops.

## 2026-04-27 - [Compose @Immutable Annotation] **Learning:** Using `PersistentList` or `ImmutableSet` in StateFlow states inside Compose doesn't fully signal to the compiler to skip recomposition when the list instance hasn't changed. **Action:** Always annotate the parent UI state data class with `@Immutable` to prevent redundant UI recompositions and ensure maximum Compose performance.

## 2026-07-01 - Optimize MergeManga computation and `@Immutable` screen states in Stats
**Learning:** In `StatsViewModel`, checking for favorited manga in `mergedMangaList` used a linear search (`firstOrNull`) on a list for every item, leading to `O(N * M)` CPU overhead, combined with a duplicate `groupBy` computation that was completely discarded. Furthermore, passing `StatsConstants` UI states without `@Immutable` prevented Compose from skipping recomposition for stats screens.
**Action:** Convert the favorited list to a `Set` for `O(1)` lookups, remove the duplicate `groupBy` computation, and annotate all `StatsConstants` data classes with `@Immutable` to ensure efficient recompositions.

## 2026-07-02 - Scoped Coroutines in MangaViewModel and @Immutable Action state classes
**Learning:** Using global unmanaged coroutine launchers (`launchIO` / `launchUI`) inside `ViewModel` contexts (such as `MangaViewModel`) or callback lambdas bypasses structured concurrency, creating potential memory leaks and resource leaks when the ViewModel's lifecycle ends. Additionally, passing custom actions or display options data classes to Compose screens without the `@Immutable` annotation prevents Compose from marking them stable, resulting in redundant screen recompositions.
**Action:** Use `viewModelScope.launchIO`, `viewModelScope.launchUI`, or child `launch` blocks to bind coroutine lifecycles, and annotate all action and intermediate state data classes in screens and constants with `@Immutable` to maximize Compose stability.
