## 2026-03-10 - Optimize LibraryViewModel flows to prevent O(N) allocations on search **Learning:** Recomputing download counts and mapping LibraryMangaItems on every search query inside `activeMangaFlow` causes O(N) object recreation and database map lookups on every keystroke, leading to performance bottlenecks when the library is large. **Action:** Moved `downloadCountMapFlow` and `trackMapFlow` to combine with the database query in `libraryMangaListFlow` directly. This enables searching and filtering flows to just operate on the previously constructed objects, preventing unnecessary memory allocations during active typing or filtering.
## 2026-03-10 - [Overclock: Cold Flow Sharing] **Learning:** Complex Kotlin Flow pipelines using `combine` and `map` repeatedly subscribe to intermediate flows. If those flows observe a database (`asFlow()`) or do expensive processing, multiple concurrent `combine` streams will execute the query redundantly per downstream subscriber. **Action:** Use `.shareIn(scope, SharingStarted.WhileSubscribed(), 1)` on shared, expensive intermediate flows (like `categoryListFlow`, `trackMapFlow`, and `filteredMangaListFlow`) to ensure they only spin up their heavy operation once, sharing the cached latest value to all parallel collectors.
## 2026-03-10 - [Overclock: Backup Restoration N+1 Optimization] **Learning:** Iterating over massive nested lists of `BackupHistory` objects inside a database transaction and individually querying the database using `getHistoryByChapterUrl` for each item creates a severe N+1 problem. This causes intense CPU and I/O thrashing since thousands of items run synchronous read queries within the same transaction lock. **Action:** Pre-fetch bulk data using `getHistoryByMangaId(mangaId)` and `getChapters(mangaId)` into an in-memory map prior to the loop. This reduces queries from O(N) down to O(1) for data retrieval, keeping the transaction quick and resolving the bottleneck.
## 2026-03-15 - [Library Flow Optimization] **Learning:** Unconditional Flow execution of heavy `MAX()` / `JOIN` SQL queries (like `getLastReadManga()` / `getLastFetchedManga()`) inside active `combine` ViewModels blocks the UI initialization for up to 10 seconds on large libraries (1000+ items). **Action:** Next time, wrap heavy aggregation queries inside `flatMapLatest` combined with active UI viewing preferences to ensure these queries are completely bypassed and return empty structures when the user is not actively sorting by those metrics.
## 2026-03-15 - [Overclock: Backup Restoration N+1 Optimization II]
**Learning:** During backup restoration, the previous N+1 database querying within a batch chunk of manga caused an extreme CPU bottleneck and slow transaction times. The helper `restoreChaptersForMangaOffline` was finding existing chapters within a list using an O(N²) comparison `dbChapters.find { it.url == chapter.url }`, resulting in exponential performance decay for large manga.
**Action:** Use `.associateBy { it.url }` instead of list iteration when finding intersecting chapter matches. Additionally, fetch chunk bulk data globally for the batch array before stepping into individual `writeMangaToDb` calls inside the transaction, returning modified/inserted elements forward to the history restoration logic to negate extra queries on newly appended elements.
## 2026-03-16 - [Overclock: Hot Flow Combine Blocking Optimization]
**Learning:** Performing a blocking database query like `db.getManga(mangaId).executeAsBlocking()` inside a `combine` operator of a hot flow (like `allChapterFlow`) forces the reactive pipeline to block the thread on every emission. This causes severe performance degradation, especially in complex screens observing multiple rapid changes.
**Action:** Extract the needed data into its own `mangaFlow` and pass it as an argument into the `combine` block so the pipeline becomes fully reactive, preventing synchronous database queries during state updates.
## 2026-03-19 - Decoupling High-Frequency Refresh State from Heavy Sorting **Learning:** I learned that combining high-frequency state updates like `isRefreshing` into a flow that also handles heavy DB sorting (like `sortedMangaFlow`) causes massive performance bottlenecks as it triggers unnecessary recalculations for every single item on simple status changes. **Action:** Next time, I will separate these concerns by placing the heavy sorting in a lower-frequency flow, and then layering high-frequency boolean state application in a distinct downstream `combine` or `map` step.
## 2026-05-24 - [Overclock: Decouple High-Frequency UI State from Expensive Re-sorts/Re-groups]
**Learning:** Combining high-frequency UI state elements (like collapsing/expanding categories) into the same Coroutine `combine` block as heavy data processing operations (like grouping a large list and performing O(N log N) sorting on each group) causes the app to execute massive redundant calculations on every minor UI interaction, leading to stuttering and CPU spikes.
**Action:** Always decouple distinct operational layers. Remove the fast-updating UI state (e.g., `collapsedCategories`) from the upstream flow that triggers grouping and sorting. Instead, map the heavy operation in a pure upstream flow, and apply the lightweight UI toggle (e.g., setting `isHidden = true`) in a downstream `combine` right before it reaches the View state.

## 2026-03-20 - Refactoring massive UiState data classes into distinct StateFlows
**Learning:** In Compose UI architectures, massive single `UiState` data classes cause full-screen recompositions when minor fields (like `isRefreshing`) are updated. This forces the entire layout (e.g., `MangaScreen`) to re-evaluate.
**Action:** Always refactor massive `UiState` data classes into smaller, distinct `StateFlow`s inside the ViewModel (e.g., separating `generalState` from `chapterState`). In Compose, collect each `StateFlow` separately and pass the specific state data classes to individual Composables to ensure precise and efficient recomposition scoping.

## 2026-03-23 - [Overclock: DisplayManga.resync N+1 Optimization]
**Learning:** Looping over lists of `DisplayManga` and performing individual `db.getManga(displayManga.mangaId).executeAsBlocking()` calls creates a massive N+1 bottleneck, locking up the CPU and stalling screens like Latest and Browse that handle large lists.
**Action:** Replaced individual iterations with a chunked `db.getMangas(chunk)` query to fetch all required records in a few bulk queries, mapping them by ID into memory before re-syncing properties.
## 2026-03-10 - [Overclock: Chunked Bulk Query Optimization]
**Learning:** When trying to eliminate an N+1 query loop by mapping list items to a list of IDs and performing a bulk fetch, fetching a massive list of IDs directly can hit SQLite bind parameter limits or memory pressure.
**Action:** Use `.chunked(900).flatMap { db.getChapters(it).executeAsBlocking() }` when fetching lists of items by IDs to safely handle thousands of IDs in smaller batches, preventing application crashes.
## 2026-04-12 - Fixing N+1 Queries in BackupCreator
**Learning:** When trying to fix N+1 queries by fetching dependencies for a whole list using `IN (?, ?, ...)`, passing an unbounded list of IDs directly into SQLite queries (e.g. via StorIO) can crash the app due to the `SQLITE_MAX_VARIABLE_NUMBER` limit (999 host parameters on older Android versions). It also causes memory spikes by trying to load massive sets of records at once.
**Action:** When resolving N+1 queries using an `IN` clause for collections, always process the primary list in manageable chunks (e.g., `mangaList.chunked(500)`) and execute the prefetches iteratively per chunk to stay under SQLite variable limits and reduce OOM risks.
